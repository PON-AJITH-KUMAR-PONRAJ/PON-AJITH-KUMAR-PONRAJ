In our previous blog, we delved into the features, advantages, and architecture of PostgreSQL. We explored how PostgreSQL stands out as a robust, open-source relational database management system, offering advanced features such as extensibility, reliability, flexibility, and SQL compliance. Building on that foundation, we now turn our attention to a more advanced topic of Postgres: Stored Procedure(SP).

Why should you care about stored procedures? Imagine managing a dynamic e-commerce platform with thousands of transactions occurring every minute. In such a high-stakes environment, maintaining data integrity, automating complex processes, and enhancing performance is crucial. This is where stored procedures come into play. They allow you to encapsulate complex logic within the database itself, automating tasks such as calculating discounts, updating inventory, or managing user data efficiently. By using stored procedures, you ensure that these operations are performed consistently and securely, reducing the likelihood of errors and improving overall system performance. In addition, Procedures are the backbone of any kind of trigger or transaction possible in DBMS.

Understanding this concept will enable you to harness the full power of Postgres, optimizing your database operations and enhancing the functionality of your applications. This blog will provide a comprehensive guide, from fundamental principles to practical implementations, ensuring that you gain a thorough understanding of this advanced feature.

Before we start, it’s expected that you have PostgreSQL or any other SQL application installed if you plan to code along. However, if you’re here to gain rich conceptual insights about the topics, you need not worry. Additionally, the concepts we discuss here will be equivalently useful for your examinations and interviews.

Functions and Procedures in Postgres

Function is a very important concept in the programming paradigm. It is an efficient way to encapsulate and abstract logic, allowing us to use the logic any number of times without bloating our code. Can a function be utilized in the Database Management paradigm as well?

In a DBMS (Database Management System), a function is a set of SQL statements that perform a specific task. It is stored in the database and can be called upon by its name with a set of input parameters. Functions in a DBMS are similar to functions in programming languages: they take inputs, perform an operation, and return a result.

Procedure on the other hand, though is similar to a function when it comes to encapsulating logic, or creating a reusable block of code, there are some important differences between them regarding their behavior, usage, and capabilities.

KEYNOTE 1: There will be some code snippets in this section. If you find them difficult to understand initially, don’t worry; you’ll gain a clearer grasp as you advance through the tutorial. Focus on the key concepts for now, and a detailed understanding will follow.

KEYNOTE 2: Before we dive into the specifics of PostgreSQL functions and stored procedures, it’s important to highlight that while they both encapsulate logic, they serve different purposes and have distinct characteristics. Some might find this section out of order, but it needs to be emphasized that functions and procedures are different in the context of postgres.

Detailed comparisons between Functions and Procedures:

Return Type:

  • Functions must return a value. The return type can be scalar (e.g., INTEGER, TEXT), composite (a row type), or even a set of rows (table-like structure). Functions are used when you need to return a result to the caller.
CREATE OR REPLACE FUNCTION get_employee_count()
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
  emp_count INTEGER;
BEGIN
  SELECT COUNT(*) INTO emp_count FROM employees;
  RETURN emp_count;
END;
$$;
  • Procedures do not return a value. Instead, they perform operations, such as modifying data or complex transactions, without returning results. They are used primarily for operations that don’t need to return a value to the caller.
CREATE OR REPLACE PROCEDURE add_employee(emp_name VARCHAR, emp_dept VARCHAR)
LANGUAGE plpgsql
AS $$
BEGIN
  INSERT INTO employees (name, department) VALUES (emp_name, emp_dept);
END;
$$;

Usage in SQL Statements:

  • Functions can be used within SQL queries and expressions. For example, you can use functions in SELECT, WHERE, or other SQL clauses.
SELECT get_employee_count();
  • Procedures cannot be used directly within SQL queries. They are invoked using the CALL statement and are generally used for procedural operations rather than for embedding logic in SQL queries.
CALL add_employee('John Doe', 'Engineering');

Parameters:

  • Functions can take input parameters and return a result. They can also use OUT parameters to return multiple values, but this is less common compared to returning a single result.
CREATE OR REPLACE FUNCTION get_employee_details(emp_id INTEGER)
RETURNS TABLE(name TEXT, department TEXT)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT name, department FROM employees WHERE id = emp_id;
END;
$$;
  • Procedures can take input parameters, output parameters, and INOUT parameters. This allows for more complex interaction with the calling environment.
CREATE OR REPLACE PROCEDURE get_employee_details(emp_id INTEGER, OUT emp_name TEXT, OUT emp_dept TEXT)
LANGUAGE plpgsql
AS $$
BEGIN
  SELECT name, department INTO emp_name, emp_dept FROM employees WHERE id = emp_id;
END;
$$;

Error Handling:

  • Functions handle errors with exception handling, but their primary purpose is to compute and return values.
  • Procedures also handle errors with exception handling, and they often use it to ensure transactional integrity or to manage complex operations.

Transaction Management:

  • Functions cannot manage transactions explicitly (i.e., you cannot use COMMIT or ROLLBACK inside a function). They are typically called as part of a larger transaction controlled by the calling context.
  • Procedures can manage transactions if they are called with the CALL statement. Procedures can include transaction control commands like COMMIT and ROLLBACK if needed.

Procedural Language

To write a procedure or function in PostgreSQL or any other database system, we need a language that provides syntax and rulesets to follow. This necessity brings us to the concept of a dedicated language for the task.

A Procedural Language (PL) is a type of programming language that includes control structures such as loops, conditionals, and variables, enabling more complex and procedural logic in database operations. Unlike standard SQL, which is declarative and focuses on what needs to be done, PL allows you to specify how tasks should be done.

PostgreSQL supports several procedural languages. The most commonly used are:

  1. PL/pgSQL: The native PostgreSQL procedural language similar to PL/SQL in Oracle.
  2. PL/Tcl: Integrates Tcl scripting language with PostgreSQL.
  3. PL/Perl: Integrates Perl scripting language with PostgreSQL.
  4. PL/Python: Integrates Python scripting language with PostgreSQL.
  5. PLV8: Integrates JavaScript (V8 engine) with PostgreSQL.
  6. PL/Java: Integrates Java programming language with PostgreSQL.
  7. PL/R: Integrates R programming language with PostgreSQL.

In this tutorial, we will be sticking to PL/pgSQL as it is a powerful and versatile choice that offers seamless integration with SQL and a rich set of features for most database operations. However, you are free to use any other language that you are comfortable with.

A Block of code in Postgres

When working with Postgres, understanding the basic structure of a code block is essential for implementing procedures and functions effectively. In this section, we’ll dissect a typical PL/pgSQL code block and explain its components, which will help you grasp how to build and execute procedures or functions later in the tutorial.

Here’s a basic PL/pgSQL code block:

do
$$
declare
-- local variables
begin
-- procedures
exception
-- error handling
end;
$$ language plpgsql;

Let’s break down each part of this code snippet:

  1. DO Statement:
    The DO statement allows you to execute an anonymous code block in PL/pgSQL. It’s useful for running a block of code without creating a named function or procedure.
  2. Dollar-Quoted String ($$):
    The dollar-quoting mechanism ($$) is used to define a string literal in PostgreSQL. This is particularly helpful for writing multi-line code blocks, as it avoids the need for escaping single quotes within the code.
  3. DECLARE Block:
    This section is where you define local variables that will be used within the block. Variables declared here are only accessible within this code block.
  4. BEGIN Block:
    The BEGIN block is where the main logic of your code goes. This is where you write the procedures, operations, and logic that you want to execute.
  5. EXCEPTION Block:
    The EXCEPTION block is used for error handling. It allows you to catch and handle exceptions that occur during the execution of the BEGIN block. You can define custom error messages or perform specific actions in case of errors.
  6. END Statement:
    The END statement marks the end of the PL/pgSQL block.
  7. LANGUAGE plpgsql:
    This specifies the procedural language used for the code block. In this case, it’s plpgsql, PostgreSQL’s procedural language.

Key Points to Remember

  • Local Variables: Define variables in the DECLARE section to use them within your code block.
  • Main Logic: Write your main procedures and logic in the BEGIN block.
  • Error Handling: Use the EXCEPTION block to handle errors gracefully.
  • Execution: The DO statement allows for executing anonymous code blocks, which is useful for quick tests or administrative tasks.

To understand the structure of PL/pgSQL code blocks better, let’s examine a practical example. We’ll create a simple code block that checks if a number is a palindrome. A palindrome is a number or word that reads the same forward and backward.

Let’s follow the pattern as in the code structure snippet:

1. Declare Block:
Let’s declare the variables first.

DO
$$
DECLARE
    -- Local variables
    input_number INTEGER := 12321;  -- Change this number to test different cases
    number_str TEXT;
    reversed_str TEXT;

2. Begin Block:
Here we will write the logic of the block.

BEGIN
    -- Convert number to string
    number_str := input_number::TEXT;

    -- Reverse the string
    reversed_str := REVERSE(number_str);

    -- Check if the number is a palindrome
    IF number_str = reversed_str THEN
        RAISE NOTICE 'The number % is a palindrome.', input_number;
    ELSE
        RAISE NOTICE 'The number % is not a palindrome.', input_number;
    END IF;

3. Exception Block:
Although an exception block is not mandatory, it a good coding practice to add one to improve the standard of your code.

EXCEPTION
    WHEN others THEN
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;

4. End Block:

END;
$$
LANGUAGE plpgsql;

So the entire code looks something like this

DO
$$
DECLARE
    -- Local variables
    input_number INTEGER := 12321;  -- Change this number to test different cases
    number_str TEXT;
    reversed_str TEXT;
BEGIN
    -- Convert number to string
    number_str := input_number::TEXT;

    -- Reverse the string
    reversed_str := REVERSE(number_str);

    -- Check if the number is a palindrome
    IF number_str = reversed_str THEN
        RAISE NOTICE 'The number % is a palindrome.', input_number;
    ELSE
        RAISE NOTICE 'The number % is not a palindrome.', input_number;
    END IF;
EXCEPTION
    WHEN others THEN
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;
END;
$$
LANGUAGE plpgsql;

Functions in Action

In this section, we’ll start by breaking down the structure of a function, including how parameters are defined, how results are returned, and how to manage errors effectively. Then, we’ll put theory into practice with a practical example and assignments to solidify your understanding.

Here’s a basic PL/pgSQL function:

CREATE OR REPLACE FUNCTION function_name(parameter_1 <datatype>)
RETURNS <datatype>
LANGUAGE plpgsql AS $$
DECLARE
    -- Local variables
    result <datatype>; -- Assuming the function returns <datatype>
BEGIN
    -- Function logic
    RETURN result; -- Returning the result
EXCEPTION
    WHEN others THEN
        -- Error handling logic
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;
END;
$$;

When we compare the structure of a function to the code blocks we’ve covered so far, we find some additional elements that enhance its capabilities.

Let’s explore the new parts in detail:

  1. CREATE OR REPLACE FUNCTIONStatement:
    Defines a function with a specified name, parameters, and return type.
  2. Parameter List:
    Specifies the input parameters for the function.
  3. RETURNSClause:
    Indicates the data type of the value returned by the function.

Consider the following function that finds the last occurrence of a given character from the given string.

CREATE OR REPLACE FUNCTION find_last_occurrence(input_string TEXT, char_to_find CHAR)
RETURNS INTEGER
LANGUAGE plpgsql AS $$
DECLARE
    position INTEGER;
BEGIN
    -- Ensure the input character is a single character
    IF LENGTH(char_to_find) != 1 THEN
        RAISE EXCEPTION 'The character to find must be a single character.';
    END IF;

    -- Find the position of the last occurrence of the character
    position := LENGTH(input_string) - POSITION(char_to_find IN REVERSE(input_string)) + 1;

    -- Check if the character is not found
    IF POSITION(char_to_find IN REVERSE(input_string)) = 0 THEN
        RETURN -1; -- Return -1 if the character is not found
    END IF;

    RETURN position;
EXCEPTION
    WHEN others THEN
        -- Handle any unexpected errors
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;
END;
$$;

Now that the function is created let’s try to execute it,

SELECT find_last_occurrence('apple', 'p');

Procedures in Action

In this section, we’ll delve deeper into stored procedures, exploring their syntax and practical applications to enhance our understanding of this powerful feature.

A Stored Procedure (SP), or simply a Procedure, is a collection of SQL statements that are stored and executed on the database server. These procedures are assigned a name and can perform a wide range of operations, including data manipulation, complex calculations, and procedural tasks. By utilizing control structures, variables, and error handling, stored procedures allow us to encapsulate complex logic and enforce business rules efficiently. They are designed to streamline repetitive tasks and can be invoked from client applications, other stored procedures, or triggers.

As we proceed, let’s examine the syntax of stored procedures and also engage in some examples to deepen our grasp of their functionality and benefits.

Here’s a basic PL/pgSQL procedure skeleton:

CREATE OR REPLACE PROCEDURE procedure_name(
    IN parameter1 datatype,
    IN parameter2 datatype,
    OUT parameter3 datatype
)
LANGUAGE plpgsql AS $$
DECLARE
    -- Local variables
BEGIN
    -- Procedure logic

    -- Set the OUT parameter value
    parameter3 := some_value;
EXCEPTION
    WHEN others THEN
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;
END;
$$;

When we compare the structure of a procedure to that of a function, we find some additional elements that enhance its capabilities.

Let’s explore the new parts in detail:

  1. CREATE OR REPLACE PROCEDURE Statement
    Defines a procedure with a specified name and parameters.
  2. Parameter List:
    Specifies the input (IN), output (OUT), and input/output (INOUT) parameters for the procedure.
  • IN: Specifies input parameters that provide values to the procedure.
  • OUT: Specifies output parameters that the procedure sets to return values.
  • INOUT: Specifies parameters that provide input values and can be modified to return output values.

Consider the following procedure that finds the last occurrence of a given character from the given string. With this example, you will not only get more understanding of procedures but also will be able to compare how procedures differ from functions.

CREATE OR REPLACE PROCEDURE find_last_occurrence_proc(
    IN input_string TEXT,
    IN char_to_find CHAR(1),
    OUT last_occurrence INT
)
LANGUAGE plpgsql AS $$
BEGIN
    -- Initialize the output variable
    last_occurrence := -1;
    
    -- Loop through the string from the end to the beginning
    FOR i IN REVERSE LENGTH(input_string) .. 1 LOOP
        IF SUBSTRING(input_string FROM i FOR 1) = char_to_find THEN
            last_occurrence := i;
            EXIT;  -- Exit the loop once the last occurrence is found
        END IF;
    END LOOP;
EXCEPTION
    WHEN OTHERS THEN
        RAISE EXCEPTION 'An error occurred: %', SQLERRM;
END;
$$;

Now that the procedure is created let’s try to execute it by using a code block instead of directly executing it.

DO $$
DECLARE
    last_occurrence INT;
BEGIN
    CALL find_last_occurrence_proc('banana', 'a', last_occurrence);
    RAISE NOTICE 'The last occurrence of ''a'' in ''banana'' is at position: %', last_occurrence;
END $$;

Before we move to some other topic, let’s make a final comparison between function and procedure based on the examples that we implemented.

A Final Comparison: Function vs. Stored Procedure

Function:

  • Returns the result directly as the return value.
  • Can be used in SELECT statements.
  • Syntax for calling: SELECT find_last_occurrence('apple', 'p');

Stored Procedure:

  • Uses OUT parameters to return results.
  • Typically called using the CALL statement.
  • Syntax for calling: CALL find_last_occurrence_proc('banana', 'a', last_occurrence);

Both approaches encapsulate logic but are used in slightly different contexts. Functions are more suited for direct data retrieval and can be easily integrated into SELECT statements. Stored procedures are more suited for complex operations that might involve multiple steps or need to handle multiple output parameters.

Syntax for Managing Procedures and Functions

Before we delve into performance considerations, it’s essential to know how to manage your stored procedures and functions. PostgreSQL provides several commands to handle these database objects, such as removing, renaming, and altering them. Here’s a quick guide:

Removing a Procedure or Function: To remove a procedure or function, you can use the DROP command. Be cautious, as this action is irreversible and will permanently delete the object.

-- Remove a function
DROP FUNCTION function_name (argument_type [, argument_type ...]);

-- Remove a procedure
DROP PROCEDURE procedure_name (argument_type [, argument_type ...]);

-- EXAMPLE 
DROP FUNCTION find_last_occurrence(TEXT, CHAR);
DROP PROCEDURE find_last_occurrence_proc(TEXT, CHAR, INT);

Renaming a Procedure or Function: To rename an existing procedure or function, use the ALTER command. This operation updates the object's name but retains its definition and behavior.

-- Rename a function
ALTER FUNCTION old_function_name RENAME TO new_function_name;

-- Rename a procedure
ALTER PROCEDURE old_procedure_name RENAME TO new_procedure_name;

-- EXAMPLE 
ALTER FUNCTION find_last_occurrence(TEXT, CHAR) RENAME TO find_last_pos;
ALTER PROCEDURE find_last_occurrence_proc(TEXT, CHAR, INT) RENAME TO find_last_pos_proc;

Modifying a Procedure or Function: Direct modification of a procedure or function’s definition isn’t supported. To change the definition, you must first drop the existing object and then recreate it with the updated logic.

-- Drop and recreate a function with new logic
DROP FUNCTION function_name;
CREATE OR REPLACE FUNCTION function_name (parameters) RETURNS return_type AS $$
BEGIN
    -- New logic
END;
$$ LANGUAGE plpgsql;

Viewing Existing Procedures and Functions: You can use the \df and \do commands in psql to list existing functions and procedures.

-- List functions
\df

-- List procedures
\do

Performance Considerations for Functions and Stored Procedures

Creating efficient and maintainable functions and stored procedures is crucial for the overall performance and reliability of your PostgreSQL database applications. Below are some key performance considerations:

  1. Limit Memory Footprint: Be cautious about the memory usage of your procedures. Avoid large temporary variables or large result sets that can consume excessive memory.
  2. Data Types: Use the most appropriate and efficient data types for your parameters and variables. For example, use INTEGER instead of TEXT for numeric values.
  3. Loop Control: Minimize the use of loops when possible. If you need to loop through data, ensure that the dataset is as small as possible.
  4. Pass-by-Reference: Where appropriate, use OUT and INOUT parameters to avoid copying large datasets.
